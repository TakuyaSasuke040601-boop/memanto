# Bug Challenge #770: `/recall` silently ignores temporal helper filters

## Summary

`memanto.app.utils.temporal_helpers.build_temporal_query()` advertises temporal recall by adding `created_after` and `created_before` to a normal `/api/v2/agents/{agent_id}/recall` payload. The `/recall` request model did not declare those fields, so Pydantic silently discarded them before the route called `MemoryReadService.search_memories()`.

Impact: agents using the helper to ask for "today", "last 7 days", or any bounded recall received memories from the full history. That can resurrect stale preferences/decisions and directly degrades timeline-sensitive retrieval quality.

## Reproduction

From the repository root, run:

```bash
uv run python scripts/poc_temporal_recall_filter.py
```

On the vulnerable code, the script exits with:

```text
BUG REPRODUCED: RecallRequest dropped created_after, so /recall cannot apply the temporal filter.
```

The root cause is that this payload field was generated by the helper but ignored by the route model:

```python
{
    "query": "importer preference",
    "limit": 10,
    "created_after": "...Z",
}
```

## Fix

This PR adds `created_after` and `created_before` to `RecallRequest`, parses date-only and ISO-8601 values consistently with the dedicated temporal endpoints, and forwards the parsed bounds into `MemoryReadService.search_memories()`. Regression coverage verifies both the helper/model contract and the API filtering behavior.
